## 一、为什么做这个产品

这个项目来自我自己的求职体验。

AI 产品方向的实习岗位主要分布在小红书帖子和微信群里，没有统一的发布平台。格式完全不一致——有的帖子写了完整职责，有的只留一个邮箱，有的把信息藏在一段话里。结果就是：收藏了几十个帖子，不知道哪个值得认真准备；等想起来要投，帖子已经沉底找不到了。

更核心的问题是**简历定制成本**太高。每投一个岗位，我要打开 JD、打开简历、逐条比对、手动调整侧重点，导出 PDF，发出去，整个过程要 1–2 小时。更糟的是：即便花了这么多时间，我也不确定改到点上没有——是不是最重要的关键词都有覆盖、有没有遗漏 JD 里最在意的能力。

这种成本导致大多数人的真实做法是用同一版简历海投，而不是真正针对岗位定制。

**我想解决的问题**：让求职者从「发现岗位」到「投出定制简历」这条链路变短、变轻、可追踪。

## 二、产品假设与 MVP 边界

在动手之前，我梳理了五个核心假设，作为功能取舍的依据：

| 假设 | 验证方式 |
|---|---|
| 结构化展示能把判断岗位时间从 5 分钟缩到 30 秒 | 解析结果的字段完整度评分 |
| AI 匹配分析能让用户知道「改哪里」而不是「凭感觉改」 | 改写建议采纳率 |
| 量化版 / 脱敏版两种模式能覆盖不同投递场景 | Word 下载次数和模式分布 |
| 把状态看板嵌入产品会提高持续使用率 | 看板访问频次 vs 单次访问 |
| 坚持「不编造」原则会提升用户对 AI 输出的信任 | 用户是否直接使用生成内容（不修改）|

**MVP 主动排除了三件事，这不是能力限制，而是有意识的取舍：**

- **不做爬虫**：合规风险高、开发成本大，且与核心价值无关。手动粘贴可以验证完整链路。
- **不做账号系统**：鉴权、多用户隔离等基础设施会占用大量时间，MVP 只需要验证单用户场景下链路是否有价值。
- **不做真实简历解析**：Word 格式解析边缘情况多，内置演示简历足以验证匹配和生成链路。

**优先跑通的核心链路**：
```
导入招聘帖 → AI 结构化解析 → 岗位管理 → JD × 简历匹配 → 定制简历生成 → Word 下载 → 投递看板
```

## 三、关键产品决策

### 决策 1：解析结果全部可人工编辑

AI 会误判，尤其是地点、公司名等字段。如果强制信任 AI 输出，错误数据会流入下游，影响匹配质量。加入人工编辑环节，既是保险，也体现「AI 辅助而非 AI 决策」的产品原则。

### 决策 2：简历生成坚持「不编造」

每条改写建议都附改写理由（section / 原始表达 / 改写后 / 改写理由），AI 明确标注「仅优化表达，不编造事实」。缺少证据的能力显式标注「💡 建议回顾是否有可复用背景」，而不是帮用户编一段。

这是功能边界，也是产品信任的基础——如果 AI 帮用户编造了经历，面试被追问穿帮，产品价值归零。

### 决策 3：量化版 / 脱敏版一键切换

求职者在不同场景有不同需求：投一般岗位需要数据说话，投竞对或有保密要求的岗位需要隐去具体数字。脱敏规则固化在代码中（如 `89.19%` → `约 89%`），不需要用户手动处理。没有这个功能，脱敏场景下链路等于没有打通。

### 决策 4：看板承接行动，不只展示数据

每条岗位旁边有「生成简历」「标记已投递」「改变状态」三个快捷操作。如果看板和操作是分离的，用户会把看板当报表而不是工具，使用频率会下降。

## 四、迭代过程

### V0 · 信息聚合验证

内置 10 条覆盖 AI 初创、互联网大厂、AI 应用公司的 Mock 岗位数据（月之暗面、字节、智谱、阶跃、百度等），验证结构化 JD 展示是否有价值。

**验证结论**：结构化展示有价值，但 Mock 岗位无法替代用户真实需求，需要解决「怎么把真实帖子导进来」的问题。

### V1 · 招聘帖导入与 AI 解析

新增导入招聘帖页面，支持粘贴小红书分享文本（自动提取 xhslink URL、清除「复制后前往小红书」等推广尾巴）。调用 Claude API 结构化解析，输出完整度评分（0–100）和可信度说明。解析结果全部可人工编辑，加入重复导入检测。

**关键决策**：解析结果支持编辑，避免 AI 误判影响下游。

### V2 · JD × 简历匹配分析

新增匹配分析 Tab，输入目标岗位 JD + 演示简历，AI 输出：匹配度评分（0–100）、岗位核心关键词、简历匹配优势、能力缺口与补充建议、简历修改建议（含原文 → 改写后 + 改写理由）、面试可能追问、不建议夸大的部分。

**关键决策**：Tab 打开即预加载演示数据，不出现空状态，让访客即刻看到完整内容。

### V3 · 简历生成 + Word 导出 + 投递看板

一键生成定制简历全文，支持量化版 / 脱敏版切换，直接下载 .docx。改写前后对比展示，每条修改附改写理由。投递看板展示漏斗转化（收藏率、投递率、面试率）、岗位类型分布、公司类型分布，最近更新列表带快捷操作。新增产品 Landing Page 作为作品集入口。

`hasGeneratedResume` 字段独立追踪简历生成状态，不随投递状态变更丢失。

### 上线后 · Bug 修复与质量打磨

MVP 完成后主动走查，发现并修复了三个问题：

**Bug 1（数据链路断点）**：匹配分析的「推荐强化的经历方向」永远显示 Demo 数据，即使接入真实 Claude API 也不更新。根因是 `matchResumePrompt` 的 JSON Schema 没有包含 `recommendations` 字段，API 从不返回它，前端 fallback 始终指向 Demo 数据。修复路径：在 prompt 里补充字段定义 → 同步更新 mock 响应 → 前端 `runMatch()` 加入对应字段的覆盖逻辑。

**Bug 2（启动错误）**：每次 `npm run dev` 启动都有路由冲突报错，原因是页面从 `app/` 迁移到 `app/(app)/` 路由组后，`.next` 构建缓存保留了旧路径引用。清除缓存重启解决。

**Bug 3（边界处理）**：简历管理页的 `fetchResumes` 无错误处理，网络失败时直接崩溃。加入 try/catch，失败时静默保持空状态。

### 部署

推送到 GitHub，通过 Vercel 自动部署。选择 Vercel 的原因：Next.js 是 Vercel 自己的框架，零配置支持 App Router 和 API Routes；免费 Hobby tier 足够演示用；每次 `git push` 自动触发部署。

## 八、关于小红书爬虫——为什么要做、怎么做、为什么现在不做

### 为什么要做

当前产品要求用户手动复制帖子文本、粘贴到导入框，这一步有明显摩擦。用户发现一个帖子到把它导入产品，中间要切换 3 次 App、操作 5–6 步。如果能做到「在小红书看到帖子，一键同步到产品」，导入门槛几乎归零，也让「持续收集岗位」这个行为变得自然而不费力——产品的数据积累才能真正形成闭环。

### 基本思路

直接爬取小红书网页（服务端爬虫）的难度极高，合规风险也最大。更可行的路径是**浏览器插件**：用户在小红书网页端（xiaohongshu.com）浏览时，插件识别当前页面是否为招聘帖，提供一个「保存到岗位雷达」按钮，用户点击后插件提取页面文本内容，通过本地 API 发送给产品处理。

这个方案的核心优势是：数据采集由用户主动触发，而不是产品自动批量抓取。这一点在合规层面有本质区别——用户自愿操作自己账号下可见的公开内容，类似于浏览器书签或 Notion Web Clipper 的工作方式。

技术上，插件需要做三件事：
1. **内容识别**：判断当前页面是否为招聘帖（根据 URL pattern 或关键词）
2. **正文提取**：从小红书的动态渲染页面中获取帖子正文、图片 OCR 文字（如果招聘信息以图片形式发布）、发帖时间等元数据
3. **数据传输**：将提取的文本发送到本地运行的产品服务（或云端 API），触发现有的 AI 解析链路

### 主要难点

**难点 1：小红书页面是动态渲染的 SPA**

小红书网页端基于 React 构建，页面内容在 JS 执行后才渲染到 DOM。插件需要等待目标 DOM 节点出现（MutationObserver）才能提取正文，而不能直接读取 HTML。此外小红书会频繁更新页面结构，CSS selector 或 DOM 路径容易失效，维护成本高。

**难点 2：招聘内容经常以图片形式发布**

大量小红书招聘帖把核心 JD 信息放在图片里（截图、设计稿），纯文本提取会漏掉关键内容。解决这个问题需要在插件里集成 OCR，或者把图片 URL 传给后端用视觉模型识别。这个链路比纯文本解析复杂一个量级。

**难点 3：合规边界模糊**

即使走浏览器插件路径，小红书的用户协议也明确禁止「以任何方式抓取、复制平台内容」。产品层面可以通过「用户主动操作」来降低风险，但这个边界在实践中仍然模糊。一旦产品规模化，平台有权封禁账号或采取法律手段。这个风险在 MVP 验证阶段不值得承担。

**难点 4：账号与会话管理**

浏览器插件依附于用户登录态，只要用户登录了小红书，插件就能访问其可见内容。但如果未来想做服务端定时爬取（比如「每天自动扫描新帖子」），就需要维护账号 cookie 或 session，而小红书的登录态有效期短、设备指纹验证严格，服务端爬虫极容易触发封号。

### 为什么现在不做

有三个层次的原因，优先级从高到低：

**第一，核心价值尚未验证**。爬虫解决的是「数据导入」的效率问题，但产品真正的价值主张是「AI 解析 + 匹配 + 简历生成」这条链路。如果这条链路用户不买单，爬虫做得再好也没有意义。应该先用手动粘贴验证 AI 链路的价值，再投入工程资源降低导入门槛。

**第二，合规风险与当前阶段不匹配**。MVP 是个人作品集项目，规模小、用户少，平台不会关注。但如果在这个阶段就写入「自动爬取」的能力，一旦产品对外扩散，反而会主动引起合规风险。更稳健的做法是：先用浏览器插件（用户主动触发）替代手动粘贴，服务端定时爬取等到有明确商业化路径、能承担法务成本的时候再谈。

**第三，工程投入产出比低**。浏览器插件需要单独开发、审核、发布（Chrome Web Store 审核周期 1–3 周），并且要处理 XHS 页面结构变更带来的持续维护成本。在核心链路还有更高价值的扩展点（真实简历上传、匹配结果持久化）的情况下，这个工程量性价比不高。
