## 2. 需求背景

AI 产品、Agent 产品、Vibe Coding 等方向的招聘信息主要分布在小红书、微信群、公众号等非结构化渠道，求职者面临以下问题：

1. **信息分散且非结构化**：各帖格式不统一，难以快速判断是否值得深看
2. **简历定制成本高**：每投一个岗位都需手动对比 JD 修改简历，耗时 1–2 小时
3. **投递进度难追踪**：岗位、简历、投递状态散落在收藏夹、备忘录、Excel 三处
4. **改写质量无保障**：不清楚 JD 最看重哪些能力，改写方向模糊

---

## 3. 目标用户

### 主要用户

- **AI 方向应届求职者**：投 AI 产品 / Agent / Vibe Coding 实习，需要同时管理多个岗位
- **需要快速定制简历的求职者**：有基础简历，希望针对不同 JD 快速优化
- **小红书 / 社群渠道求职者**：习惯从非结构化渠道获取机会，缺乏信息整理工具

### 非目标用户

- 传统行业求职者（产品价值点与 AI 岗位高度绑定）
- 已有完整 ATS 工具的校招团队

## 4. 用户痛点

| 优先级 | 痛点 | 影响 |
|---|---|---|
| P0 | 招聘帖格式不统一，筛选效率低 | 遗漏好机会，浪费时间在无效岗位 |
| P0 | 不知道简历和岗位是否匹配 | 靠感觉海投，面试通过率低 |
| P0 | 每次改简历要 1–2 小时，重复劳动 | 改写频率低，往往用同一版简历投所有岗位 |
| P1 | 投递进度分散，容易忘记跟进 | 无法复盘，错过面试窗口 |
| P1 | 投竞对时担心数据泄露 | 手动改脱敏版，增加额外工作 |

## 5. 产品目标

### 核心目标（V1.0）

将求职者「从发现岗位到发出定制简历」的时间从 2–3 小时缩短到 15 分钟以内。

### 量化目标

> 以下为 V1.1 小规模用户测试阶段的验证目标，当前 MVP 主要验证功能链路完整性，暂未形成真实用户统计结果。

| 目标 | 衡量方式 |
|---|---|
| 招聘帖结构化解析成功率 > 90% | 解析完成且完整度 ≥ 60 分 / 总导入次数 |
| 匹配分析完成率 > 40% | 运行匹配 / 导入岗位总数 |
| 简历生成率 > 50% | 生成简历 / 完成匹配分析 |
| 从导入到下载 Word 平均耗时 < 15 分钟 | 操作时间戳差值 |

## 6. 核心使用流程

```
[首页 Landing Page]
  ↓ 点击"开始体验"
[岗位列表 /jobs]
  ↓ 点击任意岗位
[岗位详情 /jobs/:id]
  ├── Tab: 岗位详情（职责、要求、完整度）
  ├── Tab: 匹配分析（选简历 → 运行 AI → 查看评分/缺口/改写建议）
  ├── Tab: 生成简历（选简历 → 生成 → 量化/脱敏切换 → 下载 Word）
  └── Tab: 原始帖子
  ↓ 状态改为"已投递"
[投递看板 /dashboard]
  └── 查看漏斗转化、分布、快捷操作

[导入招聘帖 /import]
  ↓ 粘贴帖子文本（自动提取 URL）
  ↓ 填写基础信息（标题、链接、日期）
  ↓ 点击"AI 解析"（4步 loading 动画）
  ↓ 查看解析结果（可编辑所有字段）
  ↓ 点击"保存到岗位列表"
```

## 8. 详细功能说明

### F-01 · 招聘帖导入

**功能说明**：用户粘贴小红书分享文本或招聘帖正文，填写基础信息后调用 AI 解析，结构化保存到岗位列表。

**输入**：
- 标题（可选，XHS 分享文本首行自动填充）
- 链接（可选，XHS 分享文本自动提取 xhslink URL）
- 发布日期（默认今天）
- 帖子正文（必填）

**处理逻辑**：
1. 粘贴正文时：运行 `processShareText()`，自动提取 URL，清除推广尾巴（「复制后前往小红书」等），自动填充标题（取正文首行前 40 字）
2. 点击「AI 解析」：调用 `/api/parse`，使用 `parseXhsPostPrompt` 构建提示词，返回 JSON 结构化数据
3. 解析结果展示：显示 4 步 loading 动画，完成后渲染可编辑表单
4. 重复检测：检查 URL 是否已存在于岗位列表，存在则提示（允许继续保存）
5. 点击「保存」：写入 `/api/jobs`（POST），跳转至岗位列表

**输出**：结构化 Job 对象（见 `types/index.ts`）

**异常情况**：
- AI 解析失败：回退到 Mock 响应，显示示例数据，提示「当前为演示模式」
- 正文为空：按钮禁用，提示「请先填写帖子正文」
- URL 格式错误：仅显示警告（「链接格式不符，保存后无法跳转原帖」），不阻断保存

**验收标准**：
- [ ] 粘贴标准 XHS 分享格式后，URL 自动填充，推广尾巴被清除
- [ ] AI 解析完整度 ≥ 60 分的岗位比例 > 70%
- [ ] 解析结果所有字段均可人工编辑
- [ ] 重复 URL 有提示，不强制拦截
- [ ] 保存成功后跳转岗位列表，新岗位出现在列表顶部

---

### F-02 · AI 结构化解析

**功能说明**：调用可配置大模型 API，将招聘帖自由文本解析为结构化 JSON。

**输入**：帖子正文、标题、发布时间

**处理逻辑**：
1. 构建 `parseXhsPostPrompt` 提示词（含 JSON Schema 约束）
2. 调用 `callAI()`：配置了真实 API Key 时调用大模型 API；无 Key 或 Key = mock 时返回 Mock 响应
3. 解析返回 JSON，校验必填字段，填充默认值

**输出字段**：
- `isJobPost`: 是否为招聘帖（false 则提示「未识别为招聘帖，可手动填写」）
- `company`: 公司名（不确定填 unknown）
- `title`: 岗位名
- `companyType`: AI初创 / 互联网大厂 / AI应用公司 / 其他
- `jobType`: AI产品 / Agent产品 / 数据分析 / 运营 / 算法 / 设计 / Vibe Coding / 其他
- `location`: 北京 / 上海 / 深圳 / 杭州 / 广州 / 远程 / 不限 / 未提及 / 其他
- `responsibilities`: string[]
- `requirements`: string[]
- `bonusItems`: string[]
- `applyMethod`: string
- `tags`: string[]（3–5 个关键词）
- `jdText`: 一句话岗位摘要
- `completenessScore`: 0–100
- `credibilityNote`: 可信度说明

**异常情况**：
- JSON 解析失败：回退 Mock，记录错误日志
- API 超时（> 30s）：显示超时提示，建议重试
- `isJobPost = false`：提示「AI 判断该帖子可能不是招聘帖，请确认后继续」，不阻断保存

**验收标准**：
- [ ] Mock 模式下响应时间 < 200ms
- [ ] 真实 API 模式下 4 步 loading 动画与实际进度基本对应
- [ ] 解析结果中 location = '未提及' 时，界面显示为空并提示用户填写

### F-04 · 匹配分析

**功能说明**：基于岗位 JD 和选定简历，生成详细匹配报告。

**输入**：Job ID + Resume ID（演示简历 ID 为 `demo-resume-001`）

**处理逻辑**：
1. 调用 `/api/match`（POST `{ jobId, resumeId }`）
2. 服务端读取 Job JD + Resume 文本，构建 `matchResumePrompt`
3. AI 返回 JSON 匹配报告
4. 前端更新 Job 的 `matchScore` 字段（PATCH，fire-and-forget）

**输出字段**：
- `matchScore`: 0–100 整数
- `keywords`: 岗位核心关键词（5–8 个）
- `strengths`: 简历匹配优势（3–5 条，需引用简历原文）
- `gaps`: 能力缺口（2–4 条，需说明补充方向）
- `recommendations`: 强化方向（2–3 条）
- `rewriteSuggestions`: 改写建议（含 section / original / suggestion / reason）
- `interviewQuestions`: 面试可能追问（3–5 条）
- `warnings`: 不建议夸大的部分（0–3 条）

**初始状态**：Tab 打开时预加载 `DEMO_MATCH`（演示数据），避免空状态；有演示数据标识。

**验收标准**：
- [ ] 匹配分析 Tab 打开即有内容（演示数据），不出现空状态
- [ ] 重新运行匹配后演示数据标识消失
- [ ] 改写建议包含 original（原文）、suggestion（改写后）、reason（理由）三个字段
- [ ] warnings 中不出现不客观的表达（如「你的简历很差」）

---

### F-05 · 定制简历生成

**功能说明**：基于 JD 和选定简历，AI 生成完整优化版简历文本，支持 Word 下载。

**输入**：Job ID + Resume ID

**处理逻辑**：
1. 调用 `/api/generate-resume`（POST）
2. 服务端读取 JD + 简历文本 + 匹配分析结果（strengths/gaps），构建 `rewriteResumePrompt`
3. AI 生成完整简历文本
4. 服务端调用 `docx` 库生成 Word 文件，以 base64 返回
5. 前端 PATCH Job 的 `hasGeneratedResume = true`，并根据当前状态决定是否提升至「已生成简历」

**量化版 / 脱敏版**：
- 量化版：直接使用 AI 生成的原始文本
- 脱敏版：对文本运行 `desensitize()`（数字模糊化 + 公司名脱敏 + 项目名通用化）

**输出**：
- 改写对比展示（复用匹配分析的 `rewriteSuggestions`）
- 定制简历预览（`ResumeRenderer` 组件）
- Word 文件（base64，前端触发下载）

**异常情况**：
- 生成失败：返回 `select` 阶段，提示「生成失败，请重试」
- 脱敏版 Word 生成失败：单独提示，不影响量化版

**验收标准**：
- [ ] 生成完成后 `hasGeneratedResume` 标识持久化（状态改变后不丢失）
- [ ] Word 下载文件名包含岗位名和公司名
- [ ] 脱敏版下载文件名带「（脱敏版）」后缀
- [ ] 重新生成按钮可以清空上次结果重新开始

---

### F-06 · 投递看板

**功能说明**：以可视化方式展示求职进度和转化漏斗，支持快捷操作。

**数据来源**：全量 Job 列表，按 `createdAt` 过滤时间范围

**展示内容**：
- 顶部卡片：总岗位 / 已收藏 / 已生成简历 / 已投递 / 面试中 / Offer
- 转化率：收藏→投递率、投递→面试率、面试→Offer率
- 漏斗图（已收藏 → 已生成简历 → 已投递 → 面试中 → Offer）
- 岗位类型分布（横向进度条）
- 公司类型分布（横向进度条）
- 最近更新列表（最多 10 条，含快捷操作）

**快捷操作**（在列表项上）：
- 查看详情
- 生成简历（跳转 `/jobs/:id?tab=generate`）
- 标记已投递（仅对非已投递/面试中/Offer 状态显示）
- 状态下拉选择器

**时间筛选**：全部 / 近一周 / 近一个月（影响所有统计数据）

**数据口径**：每条岗位按当前状态计入对应统计维度（非累计，不重复计数）；`hasGeneratedResume` 独立统计已生成简历数

**验收标准**：
- [ ] 时间筛选切换后所有卡片数字同步更新
- [ ] 转化率分母为 0 时显示 `--` 而非 `NaN`
- [ ] 快捷状态变更后列表实时更新（乐观更新，不刷新页面）
- [ ] 「已生成简历」统计包含 `hasGeneratedResume = true` 的岗位（无论当前状态是否为「已生成简历」）

## 9. 非功能需求

### 9.1 响应速度

| 场景 | 目标 |
|---|---|
| 页面首次加载 | < 2s（本地 JSON 读取） |
| AI 解析（Mock 模式） | < 200ms |
| AI 解析（真实 API） | < 30s，超时显示提示 |
| 状态变更（PATCH） | < 500ms，乐观更新立即生效 |
| Word 生成与下载 | < 10s |

### 9.2 数据存储

- 当前 Demo 以本地 JSON / 临时文件方式模拟单用户数据存储，用于作品集演示；部署环境中的临时文件不保证长期持久化
- 产品化版本需迁移至 Supabase / Vercel Postgres / KV 等持久化存储
- 岗位与简历数据默认留存本地；配置真实大模型 API 时，系统仅在执行解析、匹配或生成时向模型服务发送必要文本
- 无账号系统，无跨设备同步

### 9.3 隐私与合规

- 不采集任何用户行为数据
- 不做绕过平台限制的爬虫
- 用户粘贴的是公开发布的招聘信息，版权风险由用户自行承担
- AI 生成内容包含免责提示（「不编造事实、建议核实后使用」）

### 9.4 可解释性

- 每条 AI 改写建议包含 `reason` 字段，用户知道为什么这样改
- 匹配分析区分「有证据的优势」和「简历中暂缺的能力」
- 演示数据有明确标识，不与真实数据混淆

### 9.5 演示稳定性

- Mock 模式保证无需 API Key 即可完整体验所有功能
- 演示岗位初始化幂等（多次操作不重复创建）
- Loading 动画与实际请求状态绑定（不依赖固定 setTimeout）

## 10. 埋点与指标

### 关键事件定义

### 指标计算

- **解析成功率** = `parse_post_success` / `parse_post_click`
- **匹配完成率** = `match_resume_success` / `save_job_success`（同 session 内）
- **简历生成率** = `generate_resume_success` / `match_resume_success`
- **Word 下载率** = `download_resume_click` / `generate_resume_success`
- **投递转化率** = `update_job_status(new=已投递)` / `update_job_status(new=已生成简历)`
